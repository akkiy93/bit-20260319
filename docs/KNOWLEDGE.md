## 核となる考え方

### AIは「自動化ツール」ではなく「優秀な後輩」
- 丸投げするのではなく、期待値を明確にして、要所で方向確認をする
- 自分の考えを十分インプットした上で、整理・構造化を任せる

### 「高速化 = 品質低下」は過去の話
- Claude Opus 4.6の性能向上により、速さと質が両立できるようになった
- ただし、正しい使い方（認識合わせ）をした場合に限る

## プチ仕様駆動開発フレームワーク（4ドキュメント構成）

| ファイル | 役割 |
|---------|------|
| PLAN.md | 音声入力で要望を言語化する |
| SPEC.md | Claudeとの認識齟齬をなくす仕様定義 |
| TODO.md | タスク管理・コンテキスト再開時に活用 |
| KNOWLEDGE.md | 学びと教訓の記録（本ファイル） |

**ポイント**：音声入力が最重要。キーボード入力では出せない情報量・スピードが得られる。

---

## 具体的な活用事例（記事より）

| 業務 | 効果 |
|------|------|
| 経費精算 | 30分以上 → 5〜10分に短縮 |
| 月末定常業務 | 稼働報告・通勤カウントなどをまとめて自動処理 |
| メール監視 | EventBridge + Bedrockでスター付きメールをSlack通知 |
| プレゼン資料 | Marp（Markdownベース）と組み合わせて作成 |
| タスク管理 | Todoistと連携してスマホ→デスクをシームレスに |

## ワークスタイルの変化

- **ターミナル並列作業**：複数ウィンドウでClaude Codeを同時実行 → 承認待ち時間を有効活用
- **VS Codeは精読・推敲・差分確認に限定**：書かせる作業はターミナルで

---

## 組織への波及効果

- 案件ナレッジをGitHubで共有 → 引き継ぎコストをほぼゼロに
- 属人化防止 + 業務継続性の向上が同時に実現

---

## 今回のプロジェクトへの活用ポイント

- 4ドキュメント構成をそのまま採用（PLAN / SPEC / TODO / KNOWLEDGE）
- 「半自動」の考え方 → テスト項目はAIが提案、人間が補完するフローで体現
- 音声入力で要望を言語化 → PLAN.mdに反映済み
- 「Claude Codeで成功した業務をWebアプリ化する」という発展の方向性も提示できる
